# RESTful UI 憲法

## 基本原則

### I. コスト効率性（個人開発）
このプロジェクトは有料サービスよりも無料・オープンソースツールを優先します。有料の依存関係やサービスは、明確なコスト便益分析で正当化されなければなりません。有料の代替案よりも無料プラン（例：Vercel無料プラン）を優先します。インフラコストは最小限またはゼロでなければなりません。理由：個人開発プロジェクトは継続的なコスト負担を避けるべきです。

### II. シンプルさ優先（YAGNI）
過度な設計や早期最適化を避けます。明示的に必要とされる場合のみ機能を実装します。複雑なパターンよりもシンプルで直接的な解決策を優先します。複雑さは具体的なユースケースで正当化されなければなりません。理由：個人プロジェクトは企業レベルの抽象化よりも保守性と迅速な反復を重視します。

### III. 個人開発の文脈
企業レベルのオーバーヘッドは不要です。明示的に必要でない限り、正式な承認プロセス、過度なドキュメント作成、コンプライアンス要件をスキップします。包括的なプロセスよりも動作するソフトウェアに焦点を当てます。理由：個人開発は官僚的な制約なしに、より迅速な反復と学習を可能にします。

### IV. 実用的なテスト
テストは十分であるべきで、網羅的である必要はありません。重要なパスとユーザー向け機能に焦点を当てます。コアユーザージャーニーにはE2Eテスト、複雑なロジックにはユニットテスト。些細なコードや手動検証の方が速い場合はテストをスキップします。理由：個人プロジェクトでは品質保証と開発速度のバランスを取ります。

### V. 言語ポリシー
`specs/`ディレクトリ配下のすべての仕様書は日本語で記述されなければなりません。この憲法も日本語で記述されます。すべての実装コード、コードコメント、インライン documentation は英語で記述されなければなりません。この分離により、仕様書は日本語話者のステークホルダーにとってアクセスしやすく、コードは国際的な開発者やツールにとって読みやすさを維持します。理由：明確な言語境界により混乱を防ぎ、ドキュメントとコードベース全体の一貫性を維持します。

### VI. デザイン一貫性
既存のデザインシステム（Svelte Material UI / SMUI）を維持し、新しいデザインシステムやUIフレームワークを導入してはいけません。新しい機能は既存のコンポーネントライブラリ（`@smui/`パッケージ）とテーマ設定（`src/theme/`）を使用して実装されなければなりません。既存のデザインパターン、レイアウト構造、スタイリングアプローチを再利用します。理由：デザインシステムの一貫性を保ち、開発コストを削減し、ユーザー体験の統一性を維持します。

## 開発制約

### 技術選択
- オープンソースでよくメンテナンスされているライブラリを優先
- 可能な限りベンダーロックインを避ける
- 無料ホスティングプランを使用（Vercel、GitHub Pagesなど）
- 外部サービス依存を最小化

### コード品質
- 型安全性のためのTypeScript strictモード
- コード品質のためのESLint（既存の設定）
- Svelte 5のベストプラクティス
- 明確な必要性なしに過度な抽象化をしない

## 開発ワークフロー

### 機能開発
1. まず動作するソリューションを実装
2. 必要に応じてリファクタリング、事前には行わない
3. 必要に応じてドキュメント化、網羅的には行わない
4. 早期にデプロイして反復

### テスト戦略
- 重要なユーザーフローのE2Eテスト
- 複雑なビジネスロジックのユニットテスト
- UIの仕上げには手動テストも許容
- 使い捨てコードやプロトタイプのテストはスキップ

## ガバナンス

この憲法はすべての開発決定を導きます。原則が競合する場合、コスト効率性とシンプルさが優先されます。修正にはこの文書の更新と依存テンプレートへの変更の伝播が必要です。すべての機能仕様と実装計画は、これらの原則への準拠を検証しなければなりません。
